# Ableton Day-One Adapter Audit

Date: 2026-06-10  
Status: Day-one audit only - no bridge code in this goal.  
Target app: Ableton Live 11.x / 12.x on Windows.  
Draft tier: **T1 with explicit partials**. The native path is a Remote Script plus OSC bridge, modelled on AbletonOSC. Gesture and keyboard replay remain fallback routes only.

## Source Baseline

- UIA probe: `audits/ableton-uia-audit-2026-06-10.md`. It found one top-level Ableton window, zero child elements, zero actionable named controls, and zero readable values. This means UIA cannot be the primary adapter path for Live's device, piano-roll, browser, or mixer surfaces.
- AbletonOSC baseline: AbletonOSC is a MIDI Remote Script that exposes Ableton Live over OSC and aims to expose the Live Object Model using the same hierarchy and naming structure. Source: [AbletonOSC README](https://github.com/ideoforms/AbletonOSC#abletonosc-control-ableton-live-with-osc).
- AbletonOSC transport and song API: listens on OSC port 11000 and replies on 11001; it exposes song methods and properties including `start_playing`, `stop_playing`, `is_playing`, metronome, tempo, session record, and arrangement controls. Source: [AbletonOSC README Song API](https://github.com/ideoforms/AbletonOSC#song-api).
- Live Object Model baseline: LOM defines Live objects, properties, functions, parent-child relationships, and the canonical roots such as `live_set` and `live_app`. Source: [Cycling '74 LOM overview](https://docs.cycling74.com/apiref/lom/).
- Remote Script setup baseline: Ableton documents third-party Remote Scripts under the User Library `Remote Scripts` folder, visible in Preferences > MIDI / Control Surface. Source: [Ableton third-party remote scripts](https://help.ableton.com/hc/en-us/articles/209072009-Installing-third-party-remote-scripts).

## Adapter Tier Decision

Ableton is **T1 candidate with declared partials**, not T2/T3, because the high-value creative state is available through Live's intended extension surface: Remote Scripts, AbletonOSC, and LOM. Device parameters, mixer sends, transport, arrangement operations, and clip notes are directly addressable by LOM or AbletonOSC. Browser/preset navigation and arbitrary drag payloads are partial until the Remote Script adds browser/payload helpers and a 20-run soak validates them. References: [Ableton Remote Scripts](https://help.ableton.com/hc/en-us/articles/209072009-Installing-third-party-remote-scripts), [AbletonOSC README](https://github.com/ideoforms/AbletonOSC), [LOM overview](https://docs.cycling74.com/apiref/lom/), `audits/ableton-uia-audit-2026-06-10.md`.

## Route Policy For First Bridge

| Route | Policy | References |
|---|---|---|
| r1 native | Remote Script + OSC bridge. Use AbletonOSC-compatible endpoints and LOM object paths for parameters, clips, transport, sends, arrangement, and state snapshots. | [AbletonOSC README](https://github.com/ideoforms/AbletonOSC), [LOM DeviceParameter](https://docs.cycling74.com/apiref/lom/deviceparameter/), [LOM Clip](https://docs.cycling74.com/apiref/lom/clip/), [LOM Song](https://docs.cycling74.com/apiref/lom/song/) |
| r2 semantic | Keep minimal UIA only for top-level window/dialog detection. Do not depend on UIA for Live device controls, piano roll, browser, or mixer internals. | `audits/ableton-uia-audit-2026-06-10.md`, [LOM overview](https://docs.cycling74.com/apiref/lom/) |
| r3 gesture | Always record gesture alongside r1. Use gesture fallback for browser searches, custom plugin/device UIs, Simpler sample drop zones, and any operation not yet surfaced through the bridge. | [LOM object list](https://docs.cycling74.com/apiref/lom/), [LOM SimplerDevice](https://docs.cycling74.com/apiref/lom/simplerdevice/) |
| r4 repair | Off by default. Ask-once repair only when r1/r2/r3 all fail. | `Marouba REMOVED_INTERNAL_DOCS Master/marouba-master-shipping-plan V2.md` Part 0 route ladder |
